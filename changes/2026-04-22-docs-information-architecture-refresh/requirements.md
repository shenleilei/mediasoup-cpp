# Documentation Information Architecture Refresh

## Problem Statement

The `docs/` tree currently mixes too many document classes at the same level:

- stable current-state entrypoints
- generated result pages
- design documents
- implementation checklists
- investigation notes
- one-off review reports
- long-term strategy docs

This creates three maintenance problems:

1. root-level document sprawl
2. multiple similar filenames that differ only by long prefixes
3. weak distinction between “current source of truth” and “supporting / historical process material”

## Goal

Restructure `docs/` so the first-level navigation is functional, not filename-driven.

A maintainer should be able to answer “which product/runtime function am I working on?” first, then enter the corresponding document subtree.

The target mental model is:

- platform/runtime
- QoS
  - uplink
  - plain-client
  - downlink
  - shared QoS references
- operations
- planning
- generated
- archive

## In Scope

- function-first information architecture for `docs/`
- moving non-generated hand-written docs into clearer subdirectories
- rewriting internal markdown links affected by those moves
- refreshing `docs/README.md` so it becomes a functional entry map rather than a long flat list
- adding directory README files where needed
- clarifying maintenance rules for future docs

## Out Of Scope

- changing generated report output paths under `docs/generated/`
- rewriting every legacy document’s full contents
- changing accepted product behavior
- reorganizing `changes/` or `specs/current/`

## Acceptance Criteria

1. `docs/` root SHALL primarily contain the top-level navigation and current generated/result entrypoints, not the full backlog of process documents.
2. Hand-written non-generated docs SHALL be grouped by functional domain before finer-grained document role.
3. The repository SHALL not rely on long flat filenames alone to distinguish document roles when function-level directory structure can express that role better.
4. Markdown links in the affected scope SHALL resolve after the reorganization.
5. `docs/README.md` SHALL explain the functional taxonomy and the recommended reading paths using the new structure.

## Non-Functional Requirements

- Keep generated and archived artifacts separate from hand-written current-state docs.
- Prefer simple directory semantics over clever naming schemes.
- Preserve existing stable entrypoint concepts even when files move.
- Minimize churn in files generated by scripts.

## Open Questions

- None. The working assumption is that generated outputs stay where scripts already expect them, while hand-written docs are reorganized around them.
