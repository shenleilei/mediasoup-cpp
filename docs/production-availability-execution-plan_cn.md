# 生产可用性落地计划（按周执行版）

本文将“生产可用性”拆为可执行的周计划，包含负责人、输入输出、验收口径，便于直接推进和周会跟踪。

## 1. 目标与范围

### 1.1 目标

- 在上线前完成 P0 基线，具备可观测、可告警、可恢复能力。
- 在上线后 2~4 周完成 P1 运维闭环，建立稳定值班与复盘机制。
- 持续推进 P2 容量与容灾演进，降低系统性风险。

### 1.2 非目标

- 不在本计划中展开具体代码实现细节。
- 不替代 `PRODUCTION_CHECKLIST.md` 的逐项核对；本计划负责节奏与验收组织。

## 2. 角色建议（可映射到实际团队）

- `Owner-Platform`：主机、网络、内核、系统服务。
- `Owner-Backend`：SFU 服务、鉴权、限流、配置与发布。
- `Owner-SRE`：监控、告警、值班、演练、复盘。
- `Owner-QA`：回归、压测、长稳测试与验收记录。
- `Owner-Security`：安全基线审查与风险签收。

## 3. 按周执行清单

## Week 0（1 天）：定义可用性目标（SLO）

| 项 | 负责人 | 输入 | 输出 | 验收口径 |
|---|---|---|---|---|
| 定义 SLO 指标 | Owner-SRE + Owner-Backend | 历史故障、业务目标 | SLO 草案（信令可用性、房间创建成功率、呼叫中断率、P1 响应时间） | 指标定义有公式、采样窗口、统计口径 |
| 绑定观测信号 | Owner-Backend | `/healthz`、`/metrics`、业务埋点现状 | 指标映射表（SLO ↔ 监控指标） | 每个 SLO 至少对应 1 个可采集指标 |
| 确认告警级别 | Owner-SRE | 现有告警规则 | P1/P2/P3 分级与升级规则 | P1 必须有升级路径与响应时限 |

## Week 1~2：P0 上线前闭环

| 项 | 负责人 | 输入 | 输出 | 验收口径 |
|---|---|---|---|---|
| 基础设施与网络 | Owner-Platform | 目标并发、机型信息 | 多队列网卡、域名、HTTPS/WSS、端口策略 | 端口联通、TLS 可用、软中断无明显瓶颈 |
| 内核参数调优 | Owner-Platform | 压测数据、默认 sysctl | sysctl 调优配置与持久化文件 | 重启后参数生效，压测下丢包/排队指标达标 |
| 安全基线 | Owner-Security + Owner-Backend | 当前鉴权/限流能力 | JWT 鉴权方案、限流策略、录制访问控制 | 未鉴权请求不可 join，限流规则可触发且可观测 |
| 稳定性基线 | Owner-Platform + Owner-SRE | 当前部署方式 | systemd、logrotate、磁盘/OOM 保护、崩溃告警 | 进程异常可自动拉起，日志与磁盘有阈值告警 |
| 测试门禁 | Owner-QA | 回归脚本、压测工具 | 上线前验收报告 | 全量回归通过 + 压测达标 + 24h 长稳通过 |

## Week 3~6：P1 上线后运营能力

| 项 | 负责人 | 输入 | 输出 | 验收口径 |
|---|---|---|---|---|
| 监控体系落地 | Owner-SRE | `deploy/monitoring` 模板 | 分层看板与节点接入清单 | 节点指标、进程指标、日志告警均可视化 |
| 告警分级与值班 | Owner-SRE | 组织值班安排 | 值班 SOP、升级链路、演练记录 | P1 告警在目标时间内确认与升级 |
| 运维流程 | Owner-Platform + Owner-Backend | 发布与配置现状 | 优雅重启流程、配置变更流程、录制归档流程 | 关键流程有 SOP、有回滚路径 |
| 月度治理机制 | Owner-SRE + Owner-QA | 告警与故障数据 | MTTA/MTTR、误报率、漏报率月报 | 每月复盘闭环率达标并有改进项跟踪 |

## 持续阶段（P2）：容量与容灾演进

| 项 | 负责人 | 输入 | 输出 | 验收口径 |
|---|---|---|---|---|
| 多节点高可用 | Owner-Platform + Owner-Backend | Redis 拓扑与风险评估 | Redis 高可用方案、节点故障演练报告 | 节点失效后服务连续性满足预期 |
| 媒体能力增强 | Owner-Backend | 业务优先级 | TURN / Dynacast / Simulcast 迭代计划 | 每项能力有上线前后指标对比 |
| 灰度与回滚 | Owner-SRE + Owner-Backend | 发布流程现状 | 分批放量策略、回滚手册、兼容窗口策略 | 灰度期间可快速回滚，且用户影响可控 |

## 4. 量化验收矩阵（必须）

| 维度 | 指标示例 | 建议目标（示例） | 数据来源 |
|---|---|---|---|
| 可用性 | 信令可用性 | >= 99.9%（按月） | 黑盒探活 + `/healthz` |
| 成功率 | 房间创建成功率 | >= 99.5%（按日） | 业务埋点 + 服务日志 |
| 稳定性 | 呼叫中断率 | <= 0.5%（按日） | 业务埋点 + 断线事件 |
| 响应效率 | P1 首次确认时间 | <= 5 分钟（按次） | Alertmanager / 值班系统 |
| 资源水位 | CPU / 内存 / 磁盘告警占比 | 连续超阈值告警在可控范围 | Prometheus |

> 注：以上目标值是模板。正式值应由业务目标、历史基线和容量评估共同确定。

## 5. 每周例会最小模板

- 上周完成：按“输出”逐项过稿（链接证据）。
- 本周计划：按“按周执行清单”更新状态（未开始/进行中/阻塞/完成）。
- 风险与阻塞：明确责任人、解除路径、预计时间点。
- 变更影响：是否影响 SLO、是否需要新增告警或回归项。
