# mediasoup-cpp 商业化路线图（Chrome-only）

本文档基于以下输入整理：

- 当前代码仓库状态
- [ROADMAP.md](./ROADMAP.md)
- [PRODUCTION_CHECKLIST.md](../operations/PRODUCTION_CHECKLIST.md)
- 当前 QoS 能力和测试覆盖

本文档不替代现有 [ROADMAP.md](./ROADMAP.md)，而是回答一个更直接的问题：

1. 如果目标是 `3 个月内投入商业化使用`，必须完成哪些工作
2. 如果目标是 `12 个月内达到业界领先水平`，还需要补哪些能力
3. 每一项“为什么要做 / 为什么可以后做”的依据是什么

## 1. 目标边界

### 1.1 商业化版本默认边界

为了把 3 个月目标做成一个可交付版本，首个商用版本建议明确限定为：

- 客户端只支持 `Chrome`
- 房间规模：`1v1` 到 `4-6` 路订阅的小房间
- 部署形态：单区域优先，双节点容灾为佳
- 录制：可选，不承诺大型归档平台能力
- 不承诺：
  - Safari / Firefox / iOS / Android
  - 大规模直播
  - E2EE
  - WHIP / WHEP
  - VP9 SVC 商用化
  - 跨区域级联

如果不先冻结边界，后续所有优先级都会失真。

### 1.2 “可商用”定义

这里的“可商用”不是“demo 能跑通”，而是满足下面四条：

- 用户在常见网络环境下能稳定打通
- 服务出现故障时可定位、可告警、可止血
- 服务具备最小安全边界，不会被轻易滥用
- 团队知道容量边界、故障模式和回滚方式

### 1.3 “业界领先”定义

这里的“业界领先”不是单指代码优雅，而是至少满足：

- 同类 SFU 场景下更强的带宽效率
- 更成熟的上下行 QoS 闭环
- 更好的大房间能力
- 更稳定的多节点扩展能力
- 更完整的可观测、调优和数据闭环

## 2. 判断标准

后续所有任务建议统一按下面六条判断是否进入当前阶段：

1. 没有它，用户能不能稳定打通
2. 没有它，出事故后能不能定位和止血
3. 没有它，会不会有明显安全或滥用风险
4. 没有它，会不会卡住后续核心能力
5. `Chrome-only` 是否已经天然降低了它的优先级
6. 它是“商用底线”，还是“行业领先加分项”

## 3. 三个月内必须完成

下面这些项，建议定义为 `M0-M3 商用必做项`。

| 项目 | 这项是什么 | 三个月是否必须 | 判断依据 | 验收标准 |
|---|---|---|---|---|
| 范围冻结 | 明确只做 Chrome、小房间、有限部署形态，不承诺大房间/直播/Safari/E2EE | 必须 | 不冻结范围，技术排期会持续膨胀 | 对外能力边界有书面定义，内部所有任务按此边界排期 |
| HTTPS + WSS + 域名 | 为信令和页面提供正式 TLS 接入 | 必须 | 浏览器权限、接入稳定性、上线合规都依赖它 | 生产环境统一走 HTTPS/WSS，裸 IP 不作为正式入口 |
| TURN | 为 NAT/企业网/防火墙环境提供中继 | 必须 | Chrome-only 不等于网络简单；无 TURN 会有一批用户直接打不通 | 在受限网络中可通过 TURN 建连，且有可观测指标 |
| WebSocket 鉴权 | join 房间前校验 JWT / appId / room 权限 | 必须 | 无鉴权无法商业化，任何人都能进任意房间 | join 链路必须经过 token 校验，过期/越权明确拒绝 |
| Origin 校验 | 只允许白名单 origin 建立 WebSocket upgrade | 必须 | 防止跨站滥用和伪前端接入 | 支持 `allowedOrigins` 配置，非法 origin 被拒绝 |
| 连接/消息限流 | 单 IP、单 socket 的连接和消息频率限制 | 必须 | 成本低、收益高，属于基础抗滥用能力 | 新连接和消息超限可观测、可拒绝、可配置 |
| `/healthz` / `/readyz` | 区分“进程活着”和“服务可接流量” | 必须 | 没有 readiness 就难以接入负载均衡和自动摘流 | health/ready 状态与 worker / Redis / 依赖状态一致 |
| 结构化日志 | 统一 JSON 日志，包含 room/peer/module 维度 | 必须 | 出事故后没有结构化日志，排查成本过高 | 日志可直接进入 Loki/ELK，能按 roomId / peerId 检索 |
| `/metrics` + 告警 | Prometheus 指标、Grafana 面板、基础告警 | 必须 | 没 metrics 就不知道服务是在忙、抖还是要挂 | 具备 rooms/peers/worker CPU/RSS/QoS 状态/丢包等核心面板 |
| systemd + logrotate + 优雅关停 | 进程托管、日志轮转、平滑退出 | 必须 | 没有这组能力，上线和重启都不稳 | SIGTERM 后停止接入新流量，最长等待后有序退出 |
| 压测和 soak test | 容量压测、24h 稳定性测试、泄漏检测 | 必须 | 商用必须知道容量边界，而不是靠猜 | 有固定机型容量数据、24h soak 结果和泄漏检查结论 |
| crash 注入 / Redis 断连 / reconnect 验证 | 明确主要故障模式下系统怎么坏、怎么恢复 | 必须 | 商用不是“不出错”，而是“出错可预期” | worker crash、Redis 断连、客户端重连都有可验证行为 |
| QoS 常态化回归 | 把 uplink/downlink 主链路测试固定成 gate | 必须 | QoS 是差异化能力，不 gate 化就会被回归侵蚀 | 关键用例进入固定执行入口并形成日常回归 |
| TURN/媒体端口/内核参数运维固化 | UDP 端口、防火墙、socket buffer、conntrack 等基础运维 | 必须 | 实时音视频问题很大一部分不在业务逻辑，而在内核和网络栈 | 目标机器 sysctl / ethtool / firewall 有成文基线并验证过 |
| 房间上限 / 配额 / 最小租户隔离 | 最少要能限制谁能用、用多少 | 建议必须 | 没有最小配额控制，很难商业化运营 | 存在 tenant/app 维度的房间上限、并发上限和审计 |
| 录制访问控制 | 录制文件鉴权和时效控制 | 取决于是否卖录制 | 如果录制对外开放，这是安全底线；如果首发不卖可后延 | 开启录制时，下载链路必须鉴权，不能裸暴露 |

## 4. 三个月内强烈建议完成

这些项不是绝对阻塞上线，但如果不做，商用版本会很脆。

| 项目 | 这项是什么 | 为什么建议提前 | 验收标准 |
|---|---|---|---|
| WebSocket ping/pong + idle timeout | 清理死连接和僵尸 session | 实时系统非常依赖连接状态准确性，成本低收益高 | 空闲连接探测和超时断开行为可配置、可观测 |
| reconnect 边界用例 | 断线重连后 session、consumer、QoS 状态一致性 | 商用最常见问题之一，不做会堆积疑难问题 | 重连后旧状态不污染新 session，测试用例固定化 |
| QoS 数据落盘 | 将 client/downlink QoS 关键数据写入 JSONL 或同类格式 | 后续参数校准和客户故障复盘会高度依赖数据 | 数据写入可开关、可按 room / peer 检索 |
| QoS dashboard | 房间/peer 维度 QoS 决策和状态面板 | 这是你项目差异化能力，可观测性必须跟上 | 至少能看到状态分布、pause/resume 频率和主要转移链路 |
| 双节点最小验证 | 两节点部署、room ownership 切换、Redis 降级 | 即使首发单区，也至少要验证故障恢复能力 | 双节点部署可运行，room 接管有明确行为 |

## 5. 三个月内可以不做

下面这些项在 `Chrome-only 商用 v1` 里可以后移，不应该阻塞首发。

| 项目 | 这项是什么 | 三个月是否必须 | 判断依据 |
|---|---|---|---|
| Safari / Firefox / iOS / Android 支持 | 多浏览器、多端兼容 | 否 | 你已经限定 `Chrome-only`，这会显著降低首发复杂度 |
| VP9 SVC 商用化 | 高级编解码 / layer 语义优化 | 否 | 首发不依赖它才能商用；属于体验增强项 |
| WHIP / WHEP | 标准化推拉流协议 | 否 | 如果首发主要服务自家前端，不是商用底线 |
| E2EE | Insertable Streams 端到端加密 | 否 | 技术复杂度高，除非首批客户明确需要，否则不该阻塞 |
| 大规模直播模式 | 1 publisher -> 大量 subscriber 的广播模式 | 否 | 直播是另一条产品线，不是小房间首发的底线 |
| 文档英文化 | 核心设计文档英文版 | 否 | 对首发商用帮助有限，可放到外部生态阶段 |

## 6. 三个月商用版建议验收口径

如果以 `2026-07` 为上线窗口，建议把验收口径收敛成下面几条：

### 6.1 功能范围

- Chrome 浏览器端稳定可用
- 支持 1v1 和小房间会议
- 支持基本录制（如果录制对外开放，则必须有访问控制）
- 支持单区正式部署，最好具备双节点兜底能力

### 6.2 稳定性

- 目标机器上完成 24h soak test
- 无明显内存泄漏 / fd 泄漏
- worker crash 有明确恢复或错误通知行为
- Redis 断连时服务降级行为可解释

### 6.3 运维

- 有 metrics、dashboard、告警
- 有日志检索
- 有优雅关停
- 有上线 checklist 和 incident 处理流程

### 6.4 安全

- HTTPS/WSS
- JWT 鉴权
- Origin 校验
- 限流
- 录制权限控制（如适用）

### 6.5 QoS

- uplink/downlink 主链路回归固定化
- 关键 downlink 用例：
  - `browser_downlink_v3.mjs`
  - `run_downlink_matrix.mjs --cases=D7,D8`
- 关键 uplink 用例：
  - `run_matrix.mjs` 默认主 gate
  - blind-spot 重点场景 targeted rerun

## 7. 十二个月内想做到“业界领先”必须补的能力

这些能力不是商用底线，但如果目标真的是“行业领先”，它们就不是可选项。

### 7.1 媒体面带宽效率

| 项目 | 这项是什么 | 十二个月是否必须 | 判断依据 |
|---|---|---|---|
| dynacast | 按真实订阅需求停发不需要的 simulcast / spatial layer | 必须 | 没 dynacast，很难在带宽效率和大房间成本上说领先 |
| worker 侧 layer pause/resume API | 让控制面真正能精细控制 worker 侧供给 | 必须 | dynacast 和高阶下行控制都依赖它 |
| 全局 bitrate allocator | 房间级、而不是单 subscriber 级别的带宽预算分配 | 必须 | 当前 downlink 主要还是 receive control，离真正领先还差这一层 |

### 7.2 大房间能力

| 项目 | 这项是什么 | 十二个月是否必须 | 判断依据 |
|---|---|---|---|
| active speaker 检测 | 动态识别最有价值的视频源 | 必须 | 没它很难做高质量大房间 |
| selective forwarding | 每个订阅者只收最有价值的几路 | 必须 | 大房间体验和成本控制都依赖它 |
| 大房间压测 | 10-20 人房间乃至更高规模的实测 | 必须 | 领先不是“有功能”，而是“在规模下可稳定运行” |

### 7.3 多节点和级联

| 项目 | 这项是什么 | 十二个月是否必须 | 判断依据 |
|---|---|---|---|
| PipeTransport 级联 | 跨节点房间媒体转发 | 视业务范围而定，但通常建议做 | 如果你要跨区域/跨节点扩张，这项迟早会成为核心能力 |
| 节点间 QoS 语义统一 | 级联后 QoS 仍然稳定、可解释 | 建议做 | 否则多节点会把 QoS 变成黑盒 |
| Redis 高可用/更成熟多节点治理 | Sentinel/连接池/容灾 | 建议做 | 生产规模上来后，单 Redis 会成为明显风险点 |

### 7.4 QoS 数据闭环

| 项目 | 这项是什么 | 十二个月是否必须 | 判断依据 |
|---|---|---|---|
| 生产 QoS 数据分析脚本 | 对 loss/RTT/jitter/state transition 进行离线分析 | 必须 | 没数据闭环，QoS 调优只能靠感觉 |
| uplink/downlink 阈值校准 | 根据生产数据调整 profile / pause-resume 参数 | 必须 | 这是从“能用”到“领先”的分界线 |
| 端到端延迟采集 | sender -> receiver 的真实延迟链路数据 | 建议做 | 行业领先一定要能量化真实体验，而不只是控制面事件 |
| QoS 可观测平台 | 结构化展示每个 peer 的决策链路和趋势 | 建议做 | 这是产品级 QoS 能力的重要证据层 |

### 7.5 高级体验能力（Chrome-only 反而更适合做）

| 项目 | 这项是什么 | 十二个月是否必须 | 判断依据 |
|---|---|---|---|
| VP9 / SVC | 更高效的多层编码与层选择 | 建议做 | 既然只考虑 Chrome，反而更适合在这条线上做差异化 |
| 超低延迟模式 | 为特定场景提供更低 E2E latency | 建议做 | 是高端体验能力，不是基础商用能力 |
| E2EE | 浏览器端端到端加密 | 视客户群决定 | 如果你瞄准强隐私行业，这会从加分项变成刚需 |

基于当前仓库和 [ROADMAP.md](./ROADMAP.md)，建议做如下重排。

### 8.1 提前到 M0-M1 的项

- Origin 校验
- 限流
- healthz / readyz

## 9. 推荐执行方式

### 9.1 如果只有 1-2 人

最现实的方式不是“把 roadmap 全部做完”，而是：

#### M0-M1

- 把基础安全、TLS、可观测、TURN、压测基线打牢

#### M1-M2

- 把 reconnect / Redis / 双节点 / soak / QoS 回归常态化做好

#### M2-M3

- 做商用灰度、tenant / quota 最小版、runbook / checklist / release 收口

达到这个状态后，再进入：

#### M3-M6

- 生产数据回收
- QoS 阈值校准
- 多 WorkerThread 负载均衡
- active speaker + selective forwarding

#### M6-M12

- dynacast
- 全局 allocator
- 级联
- 高级 QoS 与低延迟体验

## 10. 最终结论

### 10.1 三个月目标是否可行

可行，但必须满足下面两个前提：

1. 明确限定为 `Chrome-only + 小房间 + 非直播 + 有限商用范围`
2. 把“运维、安全、TURN、稳定性、回归”放在“高级媒体能力”之前

### 10.2 十二个月目标是否可行

可行，但前提是：

- M3 后有持续工程投入
- 不把“可观测/数据闭环”继续后移
- 真正进入 worker 侧改造，而不是永远停留在外围控制层

### 10.3 一句话版本

如果目标是 `3 个月内可以卖`，你最需要补的是：

- 接入安全
- TURN
- 结构化运维
- 压测与稳定性
- QoS 固定回归

如果目标是 `12 个月内行业领先`，你最终必须补的是：

- dynacast
- 全局 bitrate allocator
- active speaker + selective forwarding
- 生产数据驱动的 QoS 校准
- 多节点级联
