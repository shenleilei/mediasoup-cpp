# mediasoup-cpp 商业化执行计划（截止 `2026-07-14`）

这份文档只解决一个问题：

- 到 `2026-07-01`，必须达到“可以接付费流量”的状态
- 到 `2026-07-14`，把 `QoS` 从“能跑”补到“可观测、可回滚、可调优”的强化版

## 1. 范围冻结

首个商用版本必须先冻结边界，否则所有排期都会失真：

- 客户端只支持 `Chrome`
- 房间规模限定为 `1v1` 到 `4-6` 路订阅的小房间
- 部署形态以单区域为主，至少验证同区域双节点兜底
- 录制是可选能力，不承诺大型归档平台
- 本阶段不承诺：
  - Safari / Firefox / iOS / Android
  - 大规模直播
  - E2EE
  - WHIP / WHEP
  - VP9 SVC 商用化
  - 跨区域级联

## 待优化项（Worker 侧）

| 优化项 | 说明 | 优先级 |
|---|---|---|
| PlainTransport 客户端接入 | 面向 native 客户端（公网 IP），跳过 ICE/DTLS，直接 UDP 收发 RTP | P1 |
| UDP socket connect() | PlainTransport 对已知目标地址的 socket 调 connect()，缓存内核路由、启用 ICMP 错误反馈 | P2 |
| sendmmsg() 批量发送 | 一个 RTP 包转发给 N 个 Consumer 时用 sendmmsg() 合并 syscall，减少内核态切换开销，大房间场景收益明显 | P2 |

## 2. 两个截止点

### 2.1 `2026-07-01`：可上线

这里的“可上线”不是 demo 能打通，而是同时满足：

- 用户在常见网络环境下能稳定建连、发流、订阅、重连
- 出问题后能通过日志、指标、告警快速定位并止血
- 服务具备最小安全边界，不会被轻易滥用
- 团队知道容量上限、故障模式和回滚方式
- `QoS` 已经进入固定回归和最小可观测状态

### 2.2 `2026-07-14`：QoS 强化版

这里的目标不是继续扩功能，而是把 `QoS` 相关能力补强到：

- 决策链路可观测
- 问题房间可回放
- 参数可以校准
- 出现误判时可以快速降级，而不是整服务回滚

## 3. 优先级规则

后续所有任务统一按下面 4 条判断：

1. 没有它，`2026-07-01` 能不能上线
2. 没有它，出了事故能不能定位和止血
3. 没有它，`QoS` 会不会在线上变成黑盒
4. 它是上线阻塞项，还是 `2026-07-14` 后可以补强的证据层

## 4. `2026-07-01` 上线阻塞项

下面这些项应该被视为 `P0`。原则是先保上线底线，再保 `QoS` 差异化。

### 4.1 工作包 A：上线底座与网络接入

| 项目 | 具体怎么做 | 产出物 | 验收标准 |
|---|---|---|---|
| HTTPS + WSS + 域名 | 统一正式入口，浏览器端不再依赖裸 IP；明确 TLS 终止位置是 Nginx 还是 uWS 直出 | 生产域名、证书配置、部署文档 | 页面、信令、录制下载入口全部走正式域名；本地和生产配置清晰分离 |
| TURN 与媒体端口基线 | 部署 TURN；确认 UDP 端口、防火墙、socket buffer、conntrack、网卡队列和内核参数 | TURN 配置、网络基线文档、验证记录 | 受限网络下可以走 TURN 建连；压测期间无明显端口或内核瓶颈 |
| `/healthz` / `/readyz` | `healthz` 只回答进程活性，`readyz` 反映 worker/Redis/依赖是否可接流量 | 探活接口、LB 接入说明 | 进程异常、worker 不可用、依赖异常时 readiness 能正确摘流 |
| systemd + logrotate + 优雅关停 | 完成进程托管、日志轮转、停止接新流量、等待已有房间自然收尾 | service 文件、logrotate 配置、停机流程 | `SIGTERM` 后不会新接入；最长等待时间可配置；退出过程可观测 |

### 4.2 工作包 B：最小安全边界

| 项目 | 具体怎么做 | 产出物 | 验收标准 |
|---|---|---|---|
| WebSocket 鉴权 | `join` 前校验 `JWT/appId/room/peer` 权限，过期和越权明确拒绝 | token 设计、校验逻辑、错误码约定 | 任意用户不能直接加入任意房间；鉴权失败行为稳定可观测 |
| Origin 校验 | WebSocket upgrade 只接受白名单 origin | `allowedOrigins` 配置、拒绝日志 | 非法 origin 被拒绝，合法 origin 不受影响 |
| 连接和消息限流 | 限制单 IP 连接数、消息频率、异常 burst | 限流配置、触发日志和指标 | 超限连接和消息可拒绝、可观测、可调参 |
| 房间上限和最小租户隔离 | 加 `tenant/app` 维度的并发上限、房间上限、审计字段 | 配额配置、审计日志 | 某个租户异常不会拖垮全局；超额行为可拒绝、可追踪 |
| 录制访问控制 | 如果首发对外卖录制，下载链接必须带鉴权和时效控制 | 访问控制方案、签名或 token 设计 | 录制文件不能裸暴露；过期或越权访问明确失败 |

### 4.3 工作包 C：可观测与运维

| 项目 | 具体怎么做 | 产出物 | 验收标准 |
|---|---|---|---|
| 结构化日志 | 统一 JSON 日志字段，至少带 `roomId/peerId/sessionId/tenant/module` | 日志 schema、接入说明 | 可直接进入 Loki/ELK，并能按房间、用户、会话检索 |
| `/metrics` + dashboard + 告警 | 复用现有监控目录，把应用层、机器层、进程层、QoS 层接上统一看板 | Grafana 面板、Prometheus 规则、Alertmanager 路由 | 至少具备 room/peer/worker/CPU/RSS/QoS 状态/丢包/重连相关面板和基础告警 |
| 上线 checklist 与 incident runbook | 把灰度、回滚、扩容、摘流、值班流程写成 SOP | checklist、runbook、联系人和升级链路 | 值班同学不看代码也能按文档完成上线、回滚、摘流 |

### 4.4 工作包 D：故障模式验证

| 项目 | 具体怎么做 | 产出物 | 验收标准 |
|---|---|---|---|
| worker crash 验证 | 人工注入 worker crash，观察房间、客户端和告警行为 | 演练脚本、演练记录、恢复策略 | crash 后行为可解释，用户侧要么恢复，要么收到明确错误通知 |
| Redis 断连和降级验证 | 验证 Redis 不可用时的 degrade-to-local 或失败模式 | 故障注入记录、降级说明 | Redis 异常时不会出现静默错乱；行为有日志、有指标、有说明 |
| reconnect 边界用例 | 覆盖断线重连、旧请求延迟到达、旧统计上报晚到 | 固定化测试用例、失败判定规则 | 新 `sessionId` 接管后旧连接不会污染新状态 |
| 双节点最小验证 | 同区域两节点部署，验证 room ownership、摘流、接管和失败行为 | 双节点部署说明、演练结果 | 首发即使主要跑单区，也明确知道节点故障时系统怎么坏、怎么恢复 |

### 4.5 工作包 E：QoS 固定回归

`QoS` 是本项目的核心差异化能力，不能排到最后做。到 `2026-07-01` 前，至少要做到“进入固定 gate，不靠人工感觉判断”。

| 项目 | 具体怎么做 | 产出物 | 验收标准 |
|---|---|---|---|
| uplink 主 gate | 固定使用 `./scripts/run_qos_tests.sh` 和 `node tests/qos_harness/run_matrix.mjs` 跑默认主链路 | 主 gate 入口、报告归档、失败清单 | 主分支连续多天稳定通过；失败可精确定位到 case |
| downlink 主 gate | 固定 `browser_downlink_v3.mjs` 和 `run_downlink_matrix.mjs --cases=D7,D8` 作为主链路用例 | 下行报告、失败归档、执行说明 | 下行关键链路进入日常回归；不是“有人想起来才跑” |
| blind-spot targeted rerun | 对已知盲区场景保留 targeted rerun，不与默认主 gate 混淆 | 重点 case 清单、单独报告 | 出现回归后能快速缩小到具体盲区场景 |
| gate 分层 | 划分 `PR 快速 gate`、`每日 gate`、`夜间全量 gate` | 执行策略、触发入口 | 回归速度和覆盖度平衡，不因全量过慢而被绕过 |

### 4.6 工作包 F：QoS 状态一致性与最小可观测

这里不是做“很漂亮的平台”，而是确保上线时 `QoS` 不是黑盒。

| 项目 | 具体怎么做 | 产出物 | 验收标准 |
|---|---|---|---|
| reconnect 后 QoS 一致性 | 验证重连后 policy、override、snapshot、consumer 状态都绑定到新 `sessionId` | 场景测试、状态转换说明 | 重连后旧连接的统计和控制消息不会污染新连接 |
| QoS 数据落盘最小版 | 先做 JSONL 级别事件流，不做复杂平台；至少写 `roomId/peerId/sessionId/direction/quality/reason/loss/rtt/jitter/bitrate` | 数据 schema、开关、示例日志 | 能按房间或用户回放问题时段的关键状态变化 |
| QoS 最小 dashboard | 至少看到状态分布、pause/resume 频率、override 次数、主要状态转移链路 | 面板、查询语句、使用说明 | 值班可以在几分钟内判断问题更像 uplink、downlink、网络抖动还是策略误判 |
| QoS 止血开关 | 提供全局或租户级降级能力，如 `disable video pause`、`audio-only only`、`fallback legacy/full qos` | feature flag、配置说明、回滚流程 | 发现 QoS 误判时能快速降级，不需要整服务回滚 |

### 4.7 工作包 G：容量、soak 与发布冻结

| 项目 | 具体怎么做 | 产出物 | 验收标准 |
|---|---|---|---|
| 容量压测 | 在目标机型上测单 worker、单节点的安全容量，并区分 `QoS on/off` 开销 | 容量报告、建议上限、压测命令 | 不再靠经验估算容量；上线前有明确保守值 |
| 24h soak | 在开启 QoS、日志、指标的配置下连续运行，检查内存、fd、线程和进程稳定性 | soak 报告、资源曲线、异常记录 | 无明显内存泄漏、fd 泄漏、周期性抖动 |
| 发布冻结与灰度 | `2026-06-21` 之后只修 blocker；先灰度，再放正式流量 | 发布日历、灰度策略、回滚条件 | 发布日不做新功能；灰度失败时能按文档回滚 |

- Chrome 端可稳定入会、发流、订阅、重连
- TLS、TURN、鉴权、限流、房间配额已落地

## 8. 明确不让它阻塞首发的项

下列事项可以进入后续阶段，但不应阻塞 `2026-07-01`：

- Safari / Firefox / iOS / Android 支持
- WHIP / WHEP
- E2EE
- VP9 SVC 商用化
- 大规模直播模式
- 跨区域级联
- 大房间 selective forwarding / active speaker / 全局 bitrate allocator

## 9. 执行原则

- `QoS` 很重要，但不能挤掉最小安全边界、最小可观测和容量验证
- 所有 `QoS` 工作优先做“固定回归、状态一致性、可止血”，再做“更漂亮的平台”
- `2026-06-21` 之后不再加新范围，只允许修 blocker
- 每次线上问题都要沉淀为新的 `targeted rerun` 或离线分析样本
